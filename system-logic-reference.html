Jixels ID Cards System Logic

Reference document for rebuilding, redesigning, or adapting the ID card system.
ProjectJixels ID card registration, approval, card generation, and verification portal Generated2026-06-07 Main filesserver.js, index.html, admin.html, supabase-schema.sql RuntimeNode.js 18+, plain HTML/CSS/JavaScript frontend

1. High-Level Purpose

The system lets workers register for a Jixels ID card through a master QR link. Registrations are stored as Pending records. An admin reviews the records, approves or rejects them, and approved workers receive a downloadable/printable ID card. The QR on each approved card verifies the worker through the server, so inactive, rejected, suspended, lost, or deleted cards show as not valid.

Public registration Admin approval ID card print/download QR verification Backup and audit

2. Main Components

Public Page: index.html

  • Shows the registration form only when opened with ?master=TOKEN#apply.
  • Accepts worker details and a photo.
  • Shows a tracking/claim link after successful submission.
  • Shows verification result when opened with ?token=....
  • Shows worker claim/download view when opened with ?claim=....

Admin Page: admin.html

  • Admin login with session token.
  • Dashboard counts by status.
  • Search/filter registrations.
  • Approve, reject, mark inactive, edit, create manually.
  • Print/download approved ID cards and send worker notification links.

Server: server.js

  • Serves HTML/assets and all /api/* routes.
  • Loads configuration from .env.
  • Stores records in Supabase when configured, otherwise local JSON.
  • Generates worker IDs and verification tokens.
  • Handles login, password reset, audit, backup, restore, status changes.

Database: supabase-schema.sql

  • cards table stores workers and ID card status.
  • audit_log table stores key admin/system actions.
  • Unique constraints protect National ID, phone, email, and verification token.

3. Core User Flow

1. Master QRAdmin prints a master QR that points to /?master=TOKEN#apply.
2. RegistrationWorker scans QR and fills form: name, location, branch, National ID, phone, email, position, photo.
3. Pending SaveServer validates data, creates ID and verification token, saves status as Pending.
4. Admin ReviewAdmin logs in, checks record, edits if needed, then approves/rejects/inactivates.
5. Card + VerifyApproved card can be printed/downloaded. Its QR checks /api/verify?token=....

4. Business Rules

RuleLogic
Public registration gateA non-admin registration must include the current master token. Without it, the server returns a forbidden registration error.
Required fieldsName, location, branch, National ID, phone, email, position, and photo are required.
Email formatEmail must match a simple email pattern before saving.
Duplicate protectionPhone, email, and National ID must be unique.
Director limitOnly one Director card is allowed. The first Director gets JIX/D.
Default statusPublic records are always saved as Pending. Admin/manual records may set status.
ApprovalApproval sets status to Approved, ensures a verification token exists, and records approved-by and approved-at values.
Inactive/deleteDelete requests do not erase records. The system marks the card Inactive with a reason.
Verification privacyPublic verification returns only last 4 digits of National ID, not the full number.

5. ID Number Logic

The server builds worker IDs from the worker position.

PositionCodeExample ID
DirectorDJIX/D
One-word role, e.g. DesignerFirst 3 letters uppercaseJIX/DES/001
Multi-word role, e.g. Sales ManagerFirst letters, max 4JIX/SM/001
Empty/invalid roleGENJIX/GEN/001

The sequence number starts from the count of existing cards with the same normalized position plus one. If an ID already exists, the server keeps increasing the sequence until it finds a free ID.

10. Security and Configuration

  • Admin passwords are stored as PBKDF2 hashes with random salt.
  • Admin sessions use random bearer tokens and expire after 8 hours.
  • Login, reset, and API calls have simple rate limiting by IP/path.
  • Supabase service-role key must stay on the server only.
  • Content security policy allows the local app, QR server images, and html2canvas CDN.
  • Required deployment variables: PORT, PUBLIC_BASE_URL, ADMIN_USER, ADMIN_PASS, ADMIN_EMAIL, MASTER_TOKEN, SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY.
  • Email notifications need RESEND_API_KEY and RESET_FROM_EMAIL.
Do not put real passwords, service-role keys, or Resend API keys inside HTML files. Keep them in hosting environment variables or a private local .env.

11. Redesign Checklist for Another System

  1. Rename the brand, logo, colors, return address, phone number, and responsibility text.
  2. Decide the ID prefix and position-code rules. Replace JIX if needed.
  3. Create a new master token and regenerate/print the master QR only after the final hosted URL is ready.
  4. Keep the status workflow unless the new organization needs extra states.
  5. Update required registration fields if the new organization needs department, expiry date, emergency contact, or employee number.
  6. Update the database schema when adding fields, then update create, edit, CSV, backup, and public verification outputs.
  7. Keep QR verification server-based so lost/inactive cards can be invalidated after printing.
  8. Test the full path: master QR registration, pending save, admin approval, worker claim link, card download, QR verification, inactive verification.
